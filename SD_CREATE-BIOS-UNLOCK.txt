UGreen Model DXP4800 Plus
Create a SD card to unlock the AMI BIOS

Noteable benefits:

This will allow you to turn off the watchdog timer so
the system will not reboot when you install a different
operating system.

It will allow you to adjust the cpu fan and case fan
profiles. Disable the UGREEN splash graphic.

To get access to the UGREEN AMI BIOS you need to create
a SD card with these features:

1. Is recognized by the BIOS as bootable
2. Missing important boot files needed to boot

Synopsis:
	use sfdisk to copy NVMe partition table to file
	edit file to point to SD card device
   use sfdisk to copy file to SD card
	use ddrescue to copy NVMe partition 128 to SD card
	use ddrescue to copy NVMe partition 1 to SD card
	mount sd card
	delete all files in /EFI/debian on the SD card
	except grub.cfg
	reboot ... BIOS now shows on screen

Short description of how to do this. In order for the
BIOS to recognize the SD card as bootable, it needs the
split GRUB partition p128. The PARTUUID must also be the
same as the PARTUUID of the boot NVMe. A simple way to do 
this is to use ddrescue to copy partition p128 to the 
SD card.

Next it needs a boot partition p1. The PARTUUID of the
SD card must match the PARTUUID of the NVMe boot partition.
The easiest way to do this is to use ddrescue to copy p1
of the NVMe to the SD card.

These two partitions with matching PARTUUID's will let the
BIOS recognize the SD card as bootable.

Remove important files needed to boot. The only file on
partition 1 in /EFI/debian we keep is grub.cfg.
Remove the following files:

	BOOTX64.CFG
	fbx64.efi
	grub.am
	grubx64.efi
	mmx64.efi
	shimx64.efi

These are need to successfully boot. Since they are missing 
you will be kicked back to the AMI BIOS.

Partition table entries are not in disk order.

But fdisk shows 250069680 sectors? Remember that p128 didn't start until
sector 34. Sector 0 through 33 is for the partition table. So when it says
the last-lba=250069646 and we add 34 sectors, we get 250069680. We will 
need to do this math to create the SD partitions.

This partition table file is going to be used to create the partitions
on the SD card. Copy the file to a new file so we can edit it:

	cp nvme2n1_table.txt sdl_table.txt

DO THIS NOW! edit sdl_table.txt and change the device: /dev/nvme2n1
to be /dev/sdl Also change the partition table entries that start with
/dev/nvme2n1p??? to be /dev/sdl??? You don't want to trash the NVMe.
If you forget and load this file you will distroy your boot NVMe.

First we need a plan. The SD card will be as follows:

	p128 = split GRUB partition
	p1   = boot vfat partition containing two version of bookworm
	p2   = swap (Not really needed on the NAS)
	p3   = Debian Bookworm

Partiton p128 and p1 are required for the SD card to be recognized by
the UGreen BIOS. More on that later.

The partition table starts at 0 and is 34 sectors, meaning it
goes from 0 to 33 which is 34 sectors because we start counting
at zero and not 1. So since the partition table goes from 0 to 33
then the next partition, p128 must start at 34. Or another way, 
0 + 34 = 34. The next partition must start at 34.

p128 is easy. It is already complete in the sdl_table.txt file.
p128 starts at 34 and is 478 sectors. 34 + 478 = 512 The next partition
must start at 512.

p1 is also easy as it is 256MB in size; more than enough to hold
two operating system boot files. So p1 is done as well. p1 starts
at 512 + 524288 = 524800. The next partition must start at 524800.

p2 is the swap file. I am not going to use swap as it will actually
slow the system down, but it is nice to have if you are going to run
many programs and not be just a file server. Future proofing.

p2 details:

2GB is 2147483648 bytes
fdisk for the sd card showed a sector size of 512 bytes

2147483648/512=4194304 sectors needed for 2GB

p2 start is good at 524800. Now we add 4194304 sectors to it and
get 4719104. p3 then must start at 4719104.

We need to know the size of p3, so lets work backwards.

From the fdisk -l of the sd card the total number of sectors is
249999360 minus the partition table, with is 34 and we get:

249999360-34=249999326 this is the last-lba. We need to change:

last-lba: 250069646 to be last-lba: 249999326

Remember, the last-lba in the file is for the NVMe, we have
a SD card. Allocation space is a little different on different
media.

If we know p3 starts at 4719104 and ends at 249999326 but we
don't know the size, we have:

4719104 + x = 249999326
x=4194304

Math check:
Reserved start=0      + Reserved size=34        = 34
   p128 start=34      +     p128 size=478       = 512
     p1 start=512     +       p1 size=524288    = 524800
     p2 start=524800  +       p2 size=4194304   = 4719104
     p3 start=4719104 +       p3 size=245280222 = 249999326

Last chance to not distroy the NVMe. Make sure there are no
/dev/nvme entries in this file.

NOTE:
In order to boot the PARTUUID of partition 128 must be the same as the PARTUUID of the NVME p128.
In order to boot the PARTUUID of partition 1   must be the same as the PARTUUID of the NVME p1.
Since there are duplicate PARTUUID's we will use the unique UUID in the GRUB command line to
select which filesystem to mount. Do not confuse UUID with PARTUUID
