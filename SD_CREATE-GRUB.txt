UGreen Model DXP4800 Plus
Notes on how to change grub.cfg so it always boots.

If the PARTUUID of partition 1 is not the same as partition 1
of the NVMe, The card will not be recognized by UGreen's AMI
BIOS.

If you run:

	lsblk -o NAME,FSTYPE,LABEL,UUID,PARTUUID,MOUNTPOINT

This will show the UUID's and PARTUUID's

Notice how partition 1 and partition 128 have similar PARTUUID
numbers. Random numbers would not be this close:

Partiton 1:   xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxa01 (vfat boot)
Partiton 2:   xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxa02 (factory squashfs)
Partiton 128: xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxa80 (split-grub)

If the PARTUUID match, the system may still boot based on luck. If 
you add more external disks, or do a cold boot (power cord is removed 
and then plugged back in) device enumeration changes and SD card boot is
no longer guaranteed. A 'systemctl poweroff' does not turn the
power off. Look at the back and the board LED's are still on and 
the I2C bus is still active. To ensure booting off the SD card, change
the UUID of the SD card to something unique "12345678" Keep the 
PARTUUID the same as the NVMe. Why?

The file /boot/boot/initrd2.img or initrd.img are the initramfs
filesystems. The different names are for the stock image and the
upgraded versions. The shell script 'init' in that image now sources 
./scripts/functions When resolv_device() is called it uses blkid 
with the -l option. The -l option searches for the FIRST match of PARTUUID. 
We have 2 with the same PARTUUID. Which one comes first is not known 
and will change over time.

Here is the code:

resolve_device() {
   DEV="$1"
   
   case "$DEV" in
   LABEL=* | UUID=* | PARTLABEL=* | PARTUUID=*)
      DEV="$(blkid -l -t "$DEV" -o device)" || return 1
      ;; 
   esac
   [ -e "$DEV" ] && echo "$DEV"
}  

If we run:

	cat /proc/cmdline

We get:

BOOT_IMAGE=/boot/vmlinuz2 root=PARTUUID=xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxa01 rootwait console=tty0 overlay=/dev/mmcblk0p7 overlayfs=ext4 net.ifnames=0 biosdevname=0

In this instance "$DEV" is "PARTUUID=xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxa01" which is not unique.

So we change the UUID to be unique on the SD card.

We change grub.cfg located at /boot/EFI/debian/grub.cfg to not use
the default PARTUUID that is duplicated, but specify the unique UUID.

grub.cfg code:

set default="0"
set timeout="10"

menuentry "SD-CARD" {
	linux /boot/vmlinuz2 root=UUID=1234-5678 rootwait console=tty0 overlay=/dev/mmcblk0p7 overlayfs=ext4 net.ifnames=0 biosdevname=0
	initrd /boot/initrd2.img
}

Change root=PARTUUID=xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxa01 to be
root=UUID=1234-5678

The code above show it will accept UUID=* and pass that to blkid. This will 
ALWAYS come back with the device of your SD card. Try it:

	blkid -l -t UUID=1234-5678 -o device

Output: /dev/sdl1

If you don't use UUID you can get the behavior that your SD card's grub is loaded 
and shows on boot but blkid later resolves to /dev/nvme2n1p1 and boots your NVMe.

Don't touch the "overlay" options unless you have modified the init script to 
run on a flat filesystem. The UGreen default is an overlay filesystem.
